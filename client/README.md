# Vesper — Client

The Next.js 15 frontend for Vesper Messenger. All encryption and decryption happens here — the server only ever sees ciphertext.

---

## Stack

| Library | Purpose |
|---|---|
| Next.js 15 / React 19 | App Router, server components, streaming |
| TailwindCSS v4 | Styling — dark cyber/minimalist design system |
| Zustand | Auth state only (`useAuthStore`) |
| Socket.io client | Real-time messaging and WebRTC signaling |
| Web Crypto API | AES-GCM for media encryption, HKDF for legacy keys |
| WebRTC API | P2P audio/video calls and device sync |
| IndexedDB | Encrypted local storage for E2E state |
| Rust → WASM | Full Signal protocol in a Web Worker |
| next-intl | i18n (English + Ukrainian) |

## Key Concepts

### WASM Worker initialisation

The crypto worker is lazily initialised on the first call. All subsequent calls queue until the WASM binary is loaded and `__ready__` is received. This means the first message send may have a short delay (WASM load + Argon2id PIN unlock), but subsequent operations are fast.

```ts
// cryptoWorkerClient.ts
const wasm = {
  ratchetEncrypt: (sessionBytes, plaintext, aad) => call('ratchetEncrypt', ...),
  encryptKeyWithPin: (keyBytes, pin) => call('encryptKeyWithPin', ...),
  // ...
}
```

### IndexedDB vault

Private identity key bytes stored in IndexedDB are wrapped by a **vault key** derived from a random 32-byte secret in `localStorage`. This two-layer approach (localStorage secret → HKDF → AES-GCM → IndexedDB) means:

- Extracting IndexedDB alone is not enough to recover keys
- The vault key is regenerated per origin, per browser profile

### Ratchet concurrency

Multiple components may trigger encrypt/decrypt simultaneously (e.g. loading messages while sending one). `useE2E.ts` uses per-session `Promise` chains (`withLock`) to serialise all operations on a given ratchet session.

### Optimistic UI

`sendMessage` adds an optimistic entry to the message list immediately with `_pendingCipher` set to the ciphertext. When the server echo arrives via `onMessage`, the client matches it by ciphertext and replaces the optimistic entry with the confirmed message (with server-assigned ID).

## Development

```bash
# Build WASM first (required)
cd ../crypto/wasm
wasm-pack build --target web --out-dir ../../client/src/wasm
cd ../../client

npm install
cp .env.example .env.local
npm run dev
```

### Environment variables

| Variable | Description |
|---|---|
| `NEXT_PUBLIC_API_URL` | Backend API base URL (e.g. `http://localhost:4000/api`) |
| `NEXT_PUBLIC_SOCKET_URL` | WebSocket server URL (e.g. `http://localhost:4000`) |
| `NEXT_PUBLIC_VAPID_PUBLIC_KEY` | VAPID public key for push notification subscriptions |
| `NEXT_PUBLIC_SUPABASE_URL` | Supabase project URL (used to parse storage paths) |

### Available scripts

```bash
npm run dev       # development server with HMR
npm run build     # production build
npm run start     # serve production build
npm run lint      # ESLint
```

## i18n

Supported locales: `en` (default), `uk` (Ukrainian).

Locale is determined by a `locale` cookie set by the `LanguageSwitcher` component. Message files live in `messages/en.json` and `messages/uk.json`.

To add a new locale:
1. Add the locale code to `src/i18n/request.ts`
2. Create `messages/{locale}.json` mirroring the structure of `en.json`
3. Add a switch option to `LanguageSwitcher`

## PWA

The client includes a `manifest.json` and service worker (`public/sw.js`) for:
- Add to home screen on mobile
- Background push notifications

The service worker handles `push` events and displays notifications using the Web Push payload.
